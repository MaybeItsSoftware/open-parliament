# AGENTS.md

Operational guide for AI coding agents working in this repository. Humans are welcome to read it too — it doubles as a project tour.

## What this project is

**Open Hansard** is a Flutter mobile app for reading UK Parliamentary debates verbatim from Hansard. It pulls data from the official Parliament APIs and stores everything locally so the app works offline after first sync. Target platforms are mobile (iOS / Android) with desktop and web shells generated by Flutter for development convenience — production focus is mobile.

Public APIs consumed:

- **Members API** — `https://members-api.parliament.uk/api/Members` — MP profiles, party, portrait. Also `…/api/Location/Constituency/Search?searchText=` (name → constituency id), `…/Location/Constituency/{id}/ElectionResult/latest` (latest general-election result: winner, majority, turnout, electorate, and per-candidate votes / vote share), and `…/Members/Parties/StateOfTheParties/{house}/{date}` (party composition of a house on a date, used by the hemicycle seating view and party profile). Note: the July 2024 boundary review gave every GB seat a new id, so the API holds **only the 2024 election** per current constituency — the constituency page shows the latest result, not a multi-election history.
- **Hansard API** — debate sections + speeches per sitting day.
- **Bills API** — `https://bills-api.parliament.uk/api/v1/Bills` — bill search/detail/stages/news and upcoming committee sittings, backing the Bills list/detail views and the debate-title → bill deep-link (`BillsApiService`).
- **Nominatim** (OpenStreetMap) — geocoding constituency names for the member map.
- **ONS Open Geography (ArcGIS)** — Westminster constituency and local-authority-district boundary polygons for the national control map (`BoundaryApiService`).
- **OpenCouncilData** — two feeds. `https://opencouncildata.co.uk/councils.php` HTML table scraped for each council's political control (party / coalition / NOC); `parseCouncils` parses it. The `?y=<year>` parameter returns that year's historical composition (archive back to 1973), which drives the council page's control-history section; `CouncilControlService.loadCouncils({year})` caches each year in its own `councils_<year>.json` file so the current table is never overwritten. `https://opencouncildata.co.uk/csv2.php?y=<year>` is the free annual councillors CSV (name, ward, party — no contacts; those are a paid dataset); `parseCouncillors` parses it. Both cached on disk (30-day TTL) by `CouncilControlService` / `CouncillorService`. CC BY-SA 4.0, so attribute it in any UI that surfaces the data.
- **Democracy Club** (Candidates API, `candidates.democracyclub.org.uk/api/next`, CC BY 4.0) — enriches a councillor with a photo, email, social links and "first elected". There is **no person-name search**, so the join goes through ballots: `election_id=local.<slug>.<date>` lists each ward's elected people (id + name + party); we match by name to the OpenCouncilData councillor (`utils/dc_match.dart`), then fetch `people/<id>` for the photo/contacts. `CouncillorEnrichmentService` caches a per-council roster and per-person profile under `boundary_cache/dc/` (30-day TTL). Best-effort: slug/date/name mismatches just yield no enrichment, and the page falls back to OpenCouncilData-only. Attribute DC wherever its data shows.
- **parliamentlive.tv** — no documented JSON; we scrape `https://parliamentlive.tv/Search?Start=DD/MM/YYYY&End=DD/MM/YYYY` for `(guid, title)` pairs of broadcast events. Direct deep-links use `https://parliamentlive.tv/event/index/{guid}` (optional `?in=HH:MM:SS`).

## Architecture

The app follows a classic **MVVM-with-Provider** pattern, layered on top of a **local-first data service**.

```
┌──────────────────────────────────────────────────┐
│ views/                Flutter widgets (screens)  │
│  ↑ Consumer<ViewModel>                           │
│ viewmodels/           ChangeNotifier state       │
│  ↑ injected service                              │
│ services/                                        │
│   ParliamentaryDataService  ← single facade      │
│     ├── DatabaseService     (SQLite)             │
│     ├── MembersApiService   (HTTP)               │
│     └── HansardApiService   (HTTP)               │
│ models/               Plain Dart data classes    │
│ widgets/              Reusable UI building blocks│
│ utils/                Pure helpers (no I/O)      │
└──────────────────────────────────────────────────┘
```

### Data flow rules

1. **Views never talk to services directly.** A view reads a `ChangeNotifier` view-model via `Provider`/`Consumer`. The view-model owns the service call.
2. **View-models never own widgets or `BuildContext`.** They expose plain getters and `notifyListeners()` after state mutations.
3. **`ParliamentaryDataService` is the only service view-models depend on.** It internally orchestrates DB + HTTP. View-models do not import `MembersApiService` or `DatabaseService` directly.
4. **Local-first reads:** the service checks SQLite first; on miss (or stale members cache) it fetches, persists, then returns. Once a sitting day is cached it is never re-fetched.

### Caching strategy

- `members.db` — single members database, refreshed every **30 days** (`_membersCacheTtl`). Schema version is bumped via `onUpgrade` in `DatabaseService`.
- `sitting_YYYY-MM-DD.db` — one file per sitting day, **append-only after first download** (no eviction yet).
- `speaker_aliases` table — learned mapping from attribution strings (e.g. *"The Minister of State"*) to a `member_id`, scoped per date for office titles. This is what lets us resolve speakers whose `MemberId` is missing in the raw Hansard payload.

If you change DB schema, you **must** add an `onUpgrade` migration — never edit `onCreate` for an existing version.

## Coding conventions

- **Dart SDK** ≥ 3.7 (see `pubspec.yaml`). Use null safety, records, pattern matching where it improves clarity.
- **Lints** in `analysis_options.yaml` are stricter than the Flutter default — `prefer_const_*`, `always_declare_return_types`, `use_build_context_synchronously`, `avoid_print`. CI fails on any analyzer issue. Run `flutter analyze` before pushing.
- **Imports:** prefer **package-relative** (`../services/foo.dart`) within `lib/`. `avoid_relative_lib_imports` is enabled — i.e. don't `import 'package:open_hansard/...'` from inside `lib/`.
- **Comments:** match the existing style — short `///` doc comments on public types and non-obvious methods. Don't add comments that just restate the code. If the *why* isn't obvious from the name, write one line explaining the *why*.
- **Async UI:** never use a `BuildContext` after an `await` without checking `mounted` (the lint enforces this).
- **Provider:** state belongs on a `ChangeNotifier` view-model, exposed once at the top of a screen via `ChangeNotifierProvider.value(value: _vm)`. Don't push view-models above the route boundary unless the state is genuinely app-wide (`ThemeService` and `SavedSpeechesService` are, individual transcripts are not).
- **Long-press actions:** press-and-hold on a transcript speech opens `showSpeechActionsSheet` (`lib/widgets/speech_actions_sheet.dart`) to save (bookmark), copy, or share it. Sharing uses the native OS share sheet via `share_plus`; saved speeches are reviewable from Settings → Saved (`lib/views/saved_speeches_view.dart`).
- **No new dependencies without good reason.** Existing deps are listed in `pubspec.yaml`; prefer composing them.

## Testing

The suite is fast (≈1 second). Aim to add a test alongside any non-trivial change.

- **Models** — round-trip parsing (API JSON → model → DB row → model) and classifier flags (`Speech.isDivision`, `isPrayers`, `isCollectiveSpeaker`, …).
- **Services** — `MembersApiService` and `HansardApiService` are tested with an injected fake `http.Client`. Don't hit the real network in tests.
- **View-models** — covered by a `FakeParliamentaryDataService` style fake. Tests should verify: state transitions (`isLoading`, `error`, `speeches`), normalisation rules (timestamp removal, redundant-date stripping, committee roster merging), and `notifyListeners()` semantics around `dispose()`.
- **Widgets** — currently only a top-level smoke test plus a handful of targeted widget tests (`test/widgets/`). Widget tests are welcome for non-trivial new widgets, but golden tests are not yet adopted.

**Known coverage gap:** `parliamentary_data_service.dart` (the facade every view-model depends on) and the services it composes (`database_service.dart`, `boundary_service.dart`, `council_control_service.dart`, `councillor_service.dart`, `councillor_enrichment_service.dart`, `party_service.dart`) have no dedicated tests — only the lower-level `*ApiService` HTTP parsing is covered. This is the riskiest untested layer (caching TTL logic, SQLite transactions, `onUpgrade` migrations); testing it needs `sqflite_common_ffi` (or an in-memory equivalent) since `sqflite` needs a real platform channel. `party_viewmodel.dart` and `settings_viewmodel.dart` are also outside `viewmodel_test.dart`'s coverage. Worth closing before making non-trivial changes to any of these.

Run with `flutter test`. A single file: `flutter test test/viewmodels/viewmodel_test.dart`.

## Hansard data quirks (read this before touching the parser)

The Hansard XML-derived JSON has a number of edge cases that the speech pipeline already handles. If you change `Speech` or `TranscriptViewModel._normaliseSpeeches`, the following must keep working:

- **Procedural rows** without a named speaker (`memberId == null`, `attributedTo == ''`).
- **Timestamps** — rows where `ItemType == "Timestamp"` *or* the value is bare clock-time text. These are stripped from the display list and used to anchor interpolated times shown next to speeches.
- **Date headings** (`hrsTag == "hs_date"`) — usually duplicate the page title; suppressed from the display.
- **Sitting-start announcements** — *"The House met at 9.30 am"* and variants. Detected by `Speech.isSittingStartAnnouncement`; the clock time (when given numerically) is parsed by `sittingStartSeconds` and added as a free time anchor, then the speech is dropped. Debates whose only content is one of these announcements are also filtered out of the date-selector feed.
- **Committee membership blocks** — multi-row "The Committee consisted of the following Members:" preambles are merged into one synthetic procedural speech. `SpeechBlock` then renders that as a structured roster card. The merger is in `_mergeCommitteeMembershipLines`; the renderer is in `speech_block.dart` (`_CommitteeRoster`).
- **Division (vote) results** — encoded as a pipe-delimited string `index|time|ayes|noes|description|result||ayes_list|noes_list`. Detected by regex on the leading `index|HH:MM`.
- **Stage directions** — short text ending with an em-dash (`rose—`, `resumed—`) attributed to a member. Rendered as italic action rows, not full speech bubbles.
- **`[Name in the Chair]`** — extracted via regex and rendered as a divider banner with the member's portrait.
- **Collective speakers** — *Hon. Members*, *Several hon. Members rose—*, etc. Rendered with a generic group avatar.

When in doubt, classify in `Speech` (pure logic) and *render* in `SpeechBlock` (pure UI). Don't push rendering rules into the model and don't push parsing into the widget.

## Speaker resolution (the trickiest area)

Hansard often refers to ministers by their office (*"The Minister of State, Department for X"*) without a `MemberId`. `TranscriptViewModel._loadMemberProfiles` resolves these in three passes:

1. **Direct ID lookup** for any speech with a `memberId`.
2. **Cached alias lookup** in the `speaker_aliases` table. Office-title aliases are scoped by sitting date because the holder of an office changes over time.
3. **Fuzzy match** by name token overlap, optionally biased by a party hint pulled from the `(Lab)` / `(Con)` bracket in the attribution.

Newly resolved aliases are written back via `saveSpeakerAliasMemberIds` so subsequent loads are O(1).

If you touch this code, add a view-model test that pins the new behaviour — this layer is the most regression-prone in the app.

## House-style theming

Each "house" gets a brand colour reflected in the AppBar of a transcript:

- Commons / Westminster Hall — green `#006548`
- Lords / Grand Committee — red `#B50938`
- Mixed Commons & Lords — purple `#5B1A6B`

Party colours live in `lib/utils/party_colors.dart`. Always use `partyColor(...)` rather than hard-coding hex values when rendering MP-related UI; that way new parties get the fallback grey automatically.

## Working agreements for agents

- **Plan changes that span more than one file before editing.** This codebase is small but tightly layered — a change in `Speech` usually means tests, the view-model, and `SpeechBlock` all need updating.
- **Never bypass `flutter analyze` failures.** Fix the root cause; don't disable lints.
- **Run the affected test file after every meaningful change**, then `flutter test` before declaring done.
- **Don't introduce new top-level files** (READMEs, scratch docs) unless asked. Update `AGENTS.md` if you change architecture.

- **Commits:** small, focused, present-tense subject (e.g. *"Render committee roster as a structured card"*). Match the existing commit log style.

## parliamentlive.tv integration

There is **no documented JSON endpoint** that maps a Hansard sitting day to its parliamentlive.tv broadcast GUIDs. The `data.parliamentlive.tv/api/event/feed` Atom feed exists but ignores date filters and just returns the most-recent ~25 events. The only stable source is the search page itself:

```
GET https://parliamentlive.tv/Search?Start=DD/MM/YYYY&End=DD/MM/YYYY
```

Each result is rendered server-side as `<a href=".../Event/Index/{guid}"><img alt="{title}">`. `ParliamentLiveApiService.parseSearchHtml` extracts those pairs; `bestParliamentLiveMatch` (in `lib/utils/parliament_live.dart`) resolves a Hansard `Debate.title` against the event list with a normalise-then-contains heuristic that handles common variations (sitting-number suffixes, the `BSL - ` prefix, punctuation).

If you change the matcher, keep these requirements in mind:
- A Hansard committee debate often has a sitting suffix (`"Courts and Tribunals Bill (Ninth sitting)"`) while the broadcast title is bare (`"Courts and Tribunals Bill"`). The Hansard-contains-broadcast direction must match.
- A given title can appear twice on the same day (morning + afternoon committee). The current code returns the first match — disambiguating by start time is a planned Phase 2 (the `_TimeAnchor` we already harvest from "The House met at …" gives us the data to do it).
- Chamber proceedings ("Cabinet Office", "Oral Answers to Questions") have no dedicated broadcast — they're moments inside the bigger "House of Commons" video. The matcher returns null for these and the caller falls back to the day-search URL.

Deep-linking per speech via `?in=HH:MM:SS` is supported by parliamentlive.tv but not yet wired up; speech `Timecode` values are the obvious source.

## Glossary

- **Sitting** — a single day on which Parliament met. The unit of caching.
- **Debate** — a top-level item within a sitting (e.g. *Prime Minister's Questions*, *Courts and Tribunals Bill (Ninth sitting)*).
- **Contribution / speech** — a single speaker's turn within a debate.
- **Division** — a recorded vote.
- **Hansard** — the official verbatim record of UK Parliament.
